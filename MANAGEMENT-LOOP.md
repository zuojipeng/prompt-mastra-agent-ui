# 镜词 · 持续监督闭环 (Management Loop)

> 本文档定义镜词项目的监督层级、职责分工、权限边界和决策流程。
> 配合 ESCALATION-RULES.md、DAILY-STATUS.md、WEEKLY-REVIEW.md 使用。

---

## 一、监督层级图

```
┌─────────────────────────────────────────┐
│  L0 — 老板 (左继鹏)                      │
│  职责：产品方向、资源分配、最终裁决        │
│  权限：否决/转向/暂停一切任务              │
└──────────────┬──────────────────────────┘
               │ 委派 + 里程碑审阅
               ▼
┌─────────────────────────────────────────┐
│  L1 — 二把手 / 审阅官                     │
│  实体：code-review-agent 等审查 Agent     │
│  职责：代码质量、安全扫描、Spec 合规检查    │
│  权限：打回不合规代码，不参与实现决策       │
└──────────────┬──────────────────────────┘
               │ 质检报告 ← 流水线产物
               ▼
┌─────────────────────────────────────────┐
│  L2 — Hermes 总执行 (本 Agent)           │
│  模型：deepseek-v4-pro                   │
│  职责：决策、规划、排期、跨部门协调、      │
│         里程碑汇报、升级判断              │
│  权限：自主推进不请示（非升级区间内）      │
│         委派子 Agent，调配资源            │
└──────────────┬──────────────────────────┘
               │ delegate_task 委派
               ▼
┌─────────────────────────────────────────┐
│  L3 — 各子 Agent (Worker 层)             │
│  模型：flash 等轻量模型                   │
│  14+ 子 Agent：研发、测试、CR、部署、     │
│                运营、需求、UI等           │
│  职责：单一领域执行，返回可验证产出        │
│  权限：仅执行，不决策，不越级汇报          │
└─────────────────────────────────────────┘
```

## 二、各层职责详述

### L0 — 老板

| 项目         | 说明                                              |
|-------------|---------------------------------------------------|
| 输入         | 产品方向、战略意图、资源约束                         |
| 输出         | 重大决策（Pivot/Go/Stop）、里程碑确认                |
| 关注指标     | 产品能用了没、质量合不合格、进度偏不偏               |
| 不关心       | 具体技术方案、代码细节、子Agent如何调度              |
| 沟通频率     | 里程碑节点（约每 3-7 天一次）                       |
| 沟通方式     | 主动推送汇报（文件总结 + 决策待办清单）              |

### L1 — 审阅官

| 项目         | 说明                                              |
|-------------|---------------------------------------------------|
| 实体         | code-review-agent（代码+安全+Spec）                |
| 触发时机     | 每次代码提交前（研发→CR→修复→测试→合并流水线）       |
| 输出         | PASS / FAIL + 具体问题清单                          |
| 权限         | 一票否决：FAIL 时代码不得合并                      |
| 不参与       | 不参与功能设计、架构决策、排期                      |

### L2 — Hermes 总执行

| 项目         | 说明                                              |
|-------------|---------------------------------------------------|
| 输入         | 老板方向 + 目标文件 + 记忆上下文                    |
| 输出         | 拆解后委派给子Agent执行，聚合结果，向L0汇报          |
| 决策范围     | 技术方案、任务拆解、子Agent调度、质量门禁执行        |
| 不能决策     | 产品方向变更、资源重分配、里程碑取消（需L0拍板）    |
| 自主推进条件 | 单任务 ≤ 30 分钟、不涉及外部依赖变更、质量门全部通过 |
| 升级条件     | 见 ESCALATION-RULES.md                            |

### L3 — 子 Agent

| 项目         | 说明                                              |
|-------------|---------------------------------------------------|
| 输入         | L2 委派的具体任务 (goal + context)                  |
| 输出         | 可验证的结果（代码文件、测试报告、审查结论）         |
| 权限         | 在自己的工具范围内执行，不得委托他人                |
| 约束         | 不得修改项目配置、不得跳过 QA 门禁、不得越级汇报    |
| 模型         | flash 等轻量模型，降低成本                          |

## 三、决策矩阵

| 场景                        | 谁决策        | 条件                                |
|----------------------------|--------------|-------------------------------------|
| 技术选型 (库/框架/架构)      | Hermes 总执行 | 不引入新外部依赖                     |
| 引入新外部服务/API           | 升级 L0      | 涉及成本、合规、安全                 |
| 任务拆解与排期               | Hermes 总执行 | 不改变里程碑承诺                     |
| 里程碑延期                  | 升级 L0      | 超过原定日期 > 1 天                  |
| 测试未通过时是否提交         | 审阅官否决    | 强制打回，不得提交                   |
| 发现需求矛盾/模糊            | 升级 L0      | 需老板澄清                          |
| Bug 修复 vs 功能优先         | Hermes 总执行 | 影响核心流程的 Bug 优先于新功能      |
| 代码风格/重构               | Hermes 总执行 | 不影响功能的前提下自主决定           |
| 生产环境部署                | 升级 L0      | 走完 QA 后由老板最终确认             |

## 四、执行流水线 (强制)

任何功能开发必须按以下顺序，不得跳步：

```
研发 → CR 审查 → 修复 → 全量测试 → 修复 → 测试通过 → 合并 → 部署
```

关键纪律：
1. Hermes 总执行**不得自审代码**，必须委派 code-review-agent
2. 全量测试必须委派 tool-testing-agent，**不得自己跑一遍当测过了**
3. 代码合并前 `npm run qa` 必须全过
4. 部署前 L0 做最终确认

## 五、监督信号

Hermes 总执行在每个任务周期输出以下信号，供 L0 和其他层判断状态：

| 信号            | 含义                                      | 何时输出              |
|----------------|------------------------------------------|----------------------|
| 进度标记 (1/5)  | 多步骤任务中的当前进度                     | 每完成一个步骤        |
| 阻塞标记 ⛔     | 当前被某个问题卡住，需升级                 | 触发升级规则时        |
| 里程碑汇报      | 阶段性交付物准备好供审阅                   | 达到里程碑节点时      |
| 门禁结果        | QA 门禁各项的 PASS/FAIL                   | 每次跑完 QA 后        |
| 日报           | 当日完成 + 阻塞 + 明日计划                 | 每日 18:00           |
| 周报           | 本周回顾 + 偏差 + 下周重点                 | 每周日 20:00         |
