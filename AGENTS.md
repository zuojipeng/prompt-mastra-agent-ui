# 镜词 · Agent 协同职责手册

> 本文档定义镜词项目的核心 Agent、职责边界、输入输出、打回规则和交付纪律。
> Hermes 总执行负责调度和最终集成，所有子 Agent 只在自己的职责范围内产出可验证结果。

## 1. 总控原则

Hermes Orchestrator 是唯一调度者，负责把老板目标拆解成角色任务、收集产物、判断门禁、处理失败回路。

任何 Agent 遇到阻塞时，必须反馈给 Hermes，而不是自行扩大范围或跳过门禁。

标准反馈格式：

```markdown
角色：
任务：
状态：PASS / FAIL / BLOCKED
证据：
影响范围：
建议归属：
请求 Hermes 动作：
```

## 2. Agent 角色

### 2.1 Product Agent

职责：
- 定义产品问题、目标用户、使用场景和成功指标
- 拆出 MVP / P1 / P2 边界
- 维护 PRD、用户故事和验收标准
- 处理需求冲突和范围膨胀

输入：
- 老板目标
- 用户反馈
- 当前产品策略文档
- 线上或测试数据

输出：
- PRD
- 用户故事地图
- 功能优先级
- 可测试验收标准
- 风险和开放问题

禁止：
- 直接改代码
- 直接决定 UI 细节
- 写不可测试的抽象需求

打回条件：
- 成功指标不可衡量
- MVP 边界不清
- 验收标准不能转成测试用例

### 2.2 UI Agent

职责：
- 设计信息架构、页面流程、组件状态和交互规则
- 定义桌面端、移动端、加载态、空状态、错误态、成功态
- 约束页面文案和视觉层级
- 与研发确认复杂交互可实现性

输入：
- PRD
- 用户故事
- 现有页面和组件
- 设计约束

输出：
- 用户流程
- 页面清单
- 组件/状态矩阵
- 响应式规则
- 可访问性要求
- 研发交付说明

禁止：
- 设计无法落地的复杂交互
- 绕过 Product Agent 改需求
- 只给视觉方向不给状态细节

打回条件：
- 缺少错误态或加载态
- 移动端规则不明确
- 核心动作路径超过 PRD 范围

### 2.3 Engineering Agent

职责：
- 在前后端实现已通过设计门的需求
- 定义 API 契约、数据结构、校验、错误处理和兼容策略
- 保持小步提交和测试覆盖
- 遵守现有代码风格，不做无关重构

输入：
- PRD 验收标准
- UI 规格
- API / 数据契约
- 现有代码上下文

输出：
- 代码改动
- API 契约说明
- 数据迁移说明
- 本地验证结果
- 风险说明

禁止：
- 未经 Hermes 同意引入新外部依赖
- 跳过类型检查、lint、测试
- 为了赶进度修改验收标准
- 自行部署生产环境

打回条件：
- 类型检查失败
- API 字段前后端漂移
- 输入未校验
- 关键错误路径无处理
- 缺少对应测试

### 2.4 Code Review Agent

职责：
- 独立审查实现质量、行为回归、安全和测试缺口
- 只做审查，不直接实现功能
- 对 P0/P1 问题一票否决

输入：
- Diff
- PRD
- UI 规格
- 测试结果

输出：
- Findings
- 严重级别
- 文件和行号
- 是否允许进入测试阶段

重点检查：
- API 契约是否一致
- 数据兼容和迁移风险
- 错误处理是否完整
- 安全和敏感信息风险
- 测试是否覆盖核心路径

打回条件：
- 任一 P0/P1 问题未修复
- 无法证明核心流程未回归

### 2.5 Test Agent

职责：
- 把 PRD 和 UI 状态转成测试矩阵
- 执行单元、集成、E2E、smoke、回归、响应式和错误路径测试
- 输出缺陷报告和发布信心等级

输入：
- PRD 验收标准
- UI 状态矩阵
- 代码变更
- 环境信息

输出：
- 测试计划
- 测试用例矩阵
- 自动化测试结果
- 手动探索测试结果
- 缺陷报告
- Release confidence

缺陷格式：

```markdown
编号：
严重级别：
环境：
复现步骤：
期望：
实际：
证据：
建议归属：
```

打回条件：
- 核心路径失败
- E2E 失败
- 构建或类型检查失败
- 关键错误态不可恢复

### 2.6 DevOps Agent

职责：
- 管理环境、构建、部署、回滚、监控和发布说明
- 检查 Cloudflare Pages / Worker / D1 / secrets 配置
- 阻塞缺少回滚和线上验证路径的发布

输入：
- 待发布提交
- 测试报告
- 部署环境
- 配置和密钥需求

输出：
- 部署计划
- 配置清单
- 回滚计划
- 上线验证步骤
- 监控和告警清单
- 发布说明

打回条件：
- 缺少必需环境变量或 secret
- 无回滚方案
- 数据迁移不可验证
- 健康检查失败

## 3. 强制交付顺序

```text
目标输入
  -> Hermes 拆解
  -> Product Agent
  -> UI Agent
  -> Engineering Agent
  -> Code Review Agent
  -> Test Agent
  -> DevOps Agent
  -> Hermes 复盘
```

任何阶段 FAIL，都回到 Hermes，由 Hermes 判定归属并重新派发。

## 4. 提交纪律

- 每个功能切片必须能独立验证
- 提交信息必须描述行为变化
- 不允许把无关重构混入功能提交
- 不允许提交未说明的配置变更
- 不允许因构建跳过类型检查而省略单独 `tsc`

## 5. 最低质量命令

前端：

```bash
npm test
npx tsc --noEmit
npm run lint
npm run build
npm run test:smoke
```

后端：

```bash
npm test
```

涉及线上 Worker 时，还必须补充：

```bash
npx wrangler deploy --dry-run
curl /api/health
```
