# 镜词 · Agent 协同职责手册

> 本文档定义镜词项目的核心 Agent、职责边界、输入输出、打回规则和交付纪律。
> Hermes 总执行负责调度和最终集成，所有子 Agent 只在自己的职责范围内产出可验证结果。

## 1. 总控原则

Hermes Orchestrator 是唯一调度者，负责把老板目标拆解成角色任务、收集产物、判断门禁、处理失败回路。

任何 Agent 遇到阻塞时，必须反馈给 Hermes，而不是自行扩大范围或跳过门禁。

标准反馈格式：

```markdown
角色：
任务：
状态：PASS / FAIL / BLOCKED
证据：
影响范围：
建议归属：
请求 Hermes 动作：
```

## 2. Agent 角色

### 2.1 Product Agent

职责：
- 定义产品问题、目标用户、使用场景和成功指标
- 拆出 MVP / P1 / P2 边界
- 维护 PRD、用户故事和验收标准
- 处理需求冲突和范围膨胀

输入：
- 老板目标
- 用户反馈
- 当前产品策略文档
- 线上或测试数据

输出：
- PRD
- 用户故事地图
- 功能优先级
- 可测试验收标准
- 风险和开放问题

禁止：
- 直接改代码
- 直接决定 UI 细节
- 写不可测试的抽象需求

打回条件：
- 成功指标不可衡量
- MVP 边界不清
- 验收标准不能转成测试用例

### 2.2 UEAgent

职责：
- 判断产品形态、用户主路径、信息架构和交互优先级
- 定义页面流程、组件状态、响应式规则和可访问性要求
- 控制视觉层级、密度、留白、页面气质和交互反馈
- 防止功能堆叠、装饰性 UI 和不符合产品形态的设计
- 与研发确认复杂交互可实现性

输入：
- PRD
- 用户故事
- 现有页面和组件
- 设计约束
- `docs/principles/PRODUCT-TASTE.md`
- `docs/principles/UE-TASTE.md`

输出：
- 产品形态判断
- 用户流程
- 页面清单
- 组件/状态矩阵
- 响应式规则
- 可访问性要求
- 研发交付说明

禁止：
- 设计无法落地的复杂交互
- 绕过 Product Agent 改需求
- 只给视觉方向不给状态细节和主路径判断
- 为了显得丰富而增加无工作流价值的控件或装饰

打回条件：
- 缺少错误态或加载态
- 移动端规则不明确
- 核心动作路径超过 PRD 范围
- 页面气质和产品形态不匹配

详细职责见 `docs/agents/UE-AGENT.md`。

### 2.3 Architecture Agent

职责：
- 判断最小可行架构、模块边界、领域命名和依赖方向
- 识别复杂度、重复逻辑、弱契约、过度设计和后续维护风险
- 定义 API / 数据契约、测试切入点、迁移和回滚影响
- 在大功能进入研发前给出架构门禁结论

输入：
- PRD 验收标准
- UEAgent 规格
- 现有代码结构
- API / 数据契约
- `docs/principles/SOFTWARE-DESIGN.md`
- `docs/principles/AGENT-REVIEW-RUBRIC.md`

输出：
- 当前设计阅读
- 最小充分设计
- 模块和领域边界
- 被拒绝的复杂方案
- 测试策略影响
- 风险和打回建议

禁止：
- 为小功能引入框架式架构
- 忽视现有代码风格
- 用抽象掩盖不清晰的领域规则
- 把无关重构混入功能切片

打回条件：
- 外部数据边界不清或未校验
- 抽象没有真实压力支撑
- 领域命名模糊
- 代码结构会让后续迭代明显变慢

详细职责见 `docs/agents/ARCHITECTURE-AGENT.md`。

### 2.4 Engineering Agent

职责：
- 在前后端实现已通过设计门的需求
- 定义 API 契约、数据结构、校验、错误处理和兼容策略
- 保持小步提交和测试覆盖
- 遵守现有代码风格，不做无关重构

输入：
- PRD 验收标准
- UEAgent 规格
- Architecture Agent 设计结论
- API / 数据契约
- 现有代码上下文

输出：
- 代码改动
- API 契约说明
- 数据迁移说明
- 本地验证结果
- 风险说明

禁止：
- 未经 Hermes 同意引入新外部依赖
- 跳过类型检查、lint、测试
- 为了赶进度修改验收标准
- 自行部署生产环境

打回条件：
- 类型检查失败
- API 字段前后端漂移
- 输入未校验
- 关键错误路径无处理
- 缺少对应测试

### 2.5 Code Review Agent

职责：
- 独立审查实现质量、行为回归、安全和测试缺口
- 只做审查，不直接实现功能
- 对 P0/P1 问题一票否决

输入：
- Diff
- PRD
- UEAgent 规格
- 测试结果

输出：
- Findings
- 严重级别
- 文件和行号
- 是否允许进入测试阶段

重点检查：
- API 契约是否一致
- 数据兼容和迁移风险
- 错误处理是否完整
- 安全和敏感信息风险
- 测试是否覆盖核心路径

打回条件：
- 任一 P0/P1 问题未修复
- 无法证明核心流程未回归

### 2.6 Test Agent

职责：
- 把 PRD 和 UI 状态转成测试矩阵
- 执行单元、集成、E2E、smoke、回归、响应式和错误路径测试
- 输出缺陷报告和发布信心等级

输入：
- PRD 验收标准
- UE 状态矩阵
- 代码变更
- 环境信息

输出：
- 测试计划
- 测试用例矩阵
- 自动化测试结果
- 手动探索测试结果
- 缺陷报告
- Release confidence

缺陷格式：

```markdown
编号：
严重级别：
环境：
复现步骤：
期望：
实际：
证据：
建议归属：
```

打回条件：
- 核心路径失败
- E2E 失败
- 构建或类型检查失败
- 关键错误态不可恢复

### 2.7 DevOps Agent

职责：
- 管理环境、构建、部署、回滚、监控和发布说明
- 检查 Cloudflare Pages / Worker / D1 / secrets 配置
- 阻塞缺少回滚和线上验证路径的发布

输入：
- 待发布提交
- 测试报告
- 部署环境
- 配置和密钥需求

输出：
- 部署计划
- 配置清单
- 回滚计划
- 上线验证步骤
- 监控和告警清单
- 发布说明

打回条件：
- 缺少必需环境变量或 secret
- 无回滚方案
- 数据迁移不可验证
- 健康检查失败

## 3. 强制交付顺序

```text
目标输入
  -> Hermes 拆解
  -> Product Agent
  -> UEAgent
  -> Architecture Agent
  -> Engineering Agent
  -> Code Review Agent
  -> Test Agent
  -> DevOps Agent
  -> Hermes 复盘
```

任何阶段 FAIL，都回到 Hermes，由 Hermes 判定归属并重新派发。

## 4. 提交纪律

- 每个功能切片必须能独立验证
- 提交信息必须描述行为变化
- 不允许把无关重构混入功能提交
- 不允许提交未说明的配置变更
- 不允许因构建跳过类型检查而省略单独 `tsc`

## 5. 最低质量命令

前端：

```bash
npm test
npx tsc --noEmit
npm run lint
npm run build
npm run test:smoke
```

后端：

```bash
npm test
```

涉及线上 Worker 时，还必须补充：

```bash
npx wrangler deploy --dry-run
curl /api/health
```
