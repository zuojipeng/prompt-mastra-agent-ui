# 02 · User Journey

## Journey Overview

```text
1. Capture idea
2. Diagnose feasibility
3. Choose reconstruction
4. Review DirectorKit
5. Execute shots on platform
6. Record outputs
7. Submit feedback
8. Export project snapshot
9. Iterate project
```

## Decision Map

| Stage | User Question | Product Response | Primary Action |
| --- | --- | --- | --- |
| Capture | Is my idea enough? | Show required creative input and target controls | Start diagnosis |
| Diagnose | Is this idea shootable? | Score feasibility and name risks | View reconstructions |
| Reconstruct | Which direction should I use? | Compare safe, stylish, cinematic options | Choose version |
| DirectorKit | What exactly should I make? | Show story, shots, prompt, platform advice | Copy first shot prompt |
| Execute | Which shot is done? | Track pending/generated/failed/usable | Mark shot status |
| Record | Where is the generated result? | Capture link or failure note | Save note |
| Feedback | What failed and why? | Submit structured feedback | Open insight |
| Export | How do I carry this project forward? | Copy project snapshot | Paste into workspace |

## Critical Frictions

1. User may not know which section matters first.
2. Many copy actions can look equal.
3. Shot execution data can be lost because it is not a durable project entity yet.
4. Feedback insights are useful but currently feel secondary to execution.
5. Long vertical pages make it hard to keep project context visible.

## Target V3 Journey

```text
Left rail: project and stage
Center: DirectorKit decision/work surface
Right rail: execution, feedback, export
```

This keeps context, work, and next action visible at the same time.

## Success Signals

- User can identify current stage within 3 seconds.
- User can find the next primary action without reading instructions.
- User can execute one shot without scrolling through the whole DirectorKit.
- User can see progress and feedback status while reviewing shots.
- User can export a project snapshot after recording shot evidence.
