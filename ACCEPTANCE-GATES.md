# 镜词 · 阶段验收门 (Acceptance Gates)

> 每个 Gate 是项目推进的硬性检查点。不通过则打回上游重新处理，不得继续。

---

## Gate 概览

```
Gate 1           Gate 2           Gate 3           Gate 4           Gate 5
战略对齐    →    任务拆解    →    实现方案    →    测试上线    →    复盘汇报
(老板确认)       (Hermes 拆解)     (CR + 方案)       (QA 全过)         (数据 + 总结)
```

---

## Gate 1 — 战略对齐

**何时进入**：新功能/新 Sprint 启动前，Hermes 收到老板方向后。

| 项目         | 内容                                              |
|-------------|---------------------------------------------------|
| 输入         | 老板的一句话方向 / 用户反馈 / 竞品分析             |
| 必需输出     | 1. PRD 摘要（功能边界、用户价值、成功指标）          |
|              | 2. 优先级说明（P0/P1/P2 及理由）                   |
|              | 3. 与产品路线的对齐声明                             |
| 验收人       | 老板 (L0)                                         |
| 验收标准     | - 功能边界清晰，不蔓延                              |
|              | - 成功指标可量化（如：操作步骤减少 N 步）            |
|              | - 与路线图不冲突                                    |
| 不通过处理   | 打回 → Hermes 根据老板反馈重新对齐后重新提交         |
| 通过后       | 进入 Gate 2，Hermes 开始任务拆解                    |

---

## Gate 2 — 任务拆解

**何时进入**：Gate 1 通过后，Hermes 总执行开始拆解。

| 项目         | 内容                                              |
|-------------|---------------------------------------------------|
| 输入         | Gate 1 通过的 PRD 摘要                             |
| 必需输出     | 1. 工作分解清单（每项有负责人子Agent、预估时间）     |
|              | 2. 关键路径标注（哪些任务是串行瓶颈）               |
|              | 3. 风险预判（可能卡住的地方 + 备选方案）            |
|              | 4. 里程碑定义（中间交付节点）                        |
| 验收人       | Hermes 总执行自检 + 老板可随时查阅                  |
| 验收标准     | - 每项任务粒度 ≤ 30 分钟                            |
|              | - 依赖关系明确（A 完成才能 B）                      |
|              | - 每个子Agent职责单一清晰                           |
| 不通过处理   | 发现遗漏或歧义 → Hermes 重新拆解                    |
| 通过后       | 进入 Gate 3，开始具体实现                           |

---

## Gate 3 — 实现方案审查 (CR)

**何时进入**：代码编写完成后，合并到主分支前。

| 项目         | 内容                                              |
|-------------|---------------------------------------------------|
| 输入         | 完成编写的代码 + 单元测试                           |
| 必需输出     | 1. Code Review 报告（由 code-review-agent 产出）   |
|              | 2. 安全扫描结果                                    |
|              | 3. Spec 合规检查结果                                |
|              | 4. 单元测试全部通过 (vitest run)                    |
| 验收人       | code-review-agent (L1) + tool-testing-agent (L3)   |
| 验收标准     | - CR 结果 PASS                                     |
|              | - 安全扫描无高危漏洞                                |
|              | - vitest 全部通过                                  |
|              | - 无硬编码密钥/敏感信息                             |
| 不通过处理   | CR 报告中的问题 → 研发 Agent 修复 → 重新提交 CR     |
|              | 重复失败 > 2 次 → 升级 Hermes 总执行审视方案        |
| 通过后       | 进入 Gate 4，开始集成测试与上线准备                 |

---

## Gate 4 — 测试与上线准备

**何时进入**：Gate 3 通过后，代码已合并。

| 项目         | 内容                                              |
|-------------|---------------------------------------------------|
| 输入         | 合并后的主分支代码                                  |
| 必需输出     | 1. `npm run qa` 全部通过（vitest + smoke + e2e）   |
|              | 2. QA-CHECKLIST.md 全部勾选                        |
|              | 3. 浏览器手动验收报告（由 Hermes 总执行完成）        |
|              | 4. 部署就绪声明（含回滚方案）                        |
| 验收人       | tool-testing-agent (L3) → Hermes 总执行 (L2) 复验  |
| 验收标准     | - build 零错误                                     |
|              | - tsc --noEmit 零错误                              |
|              | - vitest 全部通过                                  |
|              | - smoke test 全部通过                              |
|              | - e2e-api test 全部通过                            |
|              | - 浏览器验收无阻塞问题                              |
| 不通过处理   | 任一失败 → tool-testing-agent 产出失败报告          |
|              | → 研发 Agent 修复 → 重新进入 Gate 3 (CR)            |
|              | → 重新跑全量 QA                                    |
|              | 同一问题反复出现 > 3 次 → 升级 L0                   |
| 通过后       | 通知 L0 做最终部署确认                              |

---

## Gate 5 — 复盘与汇报

**何时进入**：部署完成后，或 Sprint 结束。

| 项目         | 内容                                              |
|-------------|---------------------------------------------------|
| 输入         | 已部署的版本 + 期间数据                            |
| 必需输出     | 1. 本周期完成事项清单（含 merge/deploy 记录）       |
|              | 2. 质量数据（bug 数、测试通过率、CR 通过率）        |
|              | 3. 偏差分析（实际 vs 计划，原因 + 改进）            |
|              | 4. 经验沉淀（可复用模式、踩过的坑）                 |
|              | 5. 下周期建议（给老板的决策依据）                   |
| 验收人       | 老板 (L0)                                         |
| 验收标准     | - 每个偏差有根因分析                                |
|              | - 经验可转化为 skill 或 memory                     |
|              | - 下周期建议有优先级                                |
| 不通过处理   | 数据缺失或分析浮于表面 → Hermes 补全后重新提交      |
| 通过后       | 本周期关闭，启动新一轮 Gate 1                       |

---

## 门禁流水线总览

```
        老板方向
           │
    ┌──────▼──────┐
    │  Gate 1     │── FAIL → 重新对齐
    │  战略对齐    │
    └──────┬──────┘
           │ PASS
    ┌──────▼──────┐
    │  Gate 2     │── FAIL → 重新拆解
    │  任务拆解    │
    └──────┬──────┘
           │ PASS
    ┌──────▼──────┐
    │  Gate 3     │── FAIL → 修复后重提
    │  CR + 方案   │
    └──────┬──────┘
           │ PASS
    ┌──────▼──────┐
    │  Gate 4     │── FAIL → 修复 → 退回 Gate 3
    │  测试上线    │
    └──────┬──────┘
           │ PASS
    ┌──────▼──────┐
    │  Gate 5     │── FAIL → 补全数据
    │  复盘汇报    │
    └──────┬──────┘
           │ PASS
          🎉 本周期关闭
```

---

## 补充说明

- Gate 1 和 Gate 5 需要老板参与，其余 Gate 为 Agent 自动执行
- Gate 3→4 之间有循环回路：QA 不过会打回 CR 重新审查
- 所有 Gate 的结果记录在 `iteration-notes.md` 中
- 同一 Gate 连续失败 > 2 次，触发升级（见 ESCALATION-RULES.md）
